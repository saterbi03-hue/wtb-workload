# WU4 — WTB · Workload Backend

Moteur de calcul de charge de travail · prévision · lissage · décision.

## Files

| File | Role |
|---|---|
| `build_consolidated_data.py` | One-shot script: reads the noisy source `WTB_NEW__1_.xlsx` (14 per-CA sheets, 50+ columns each, varying layouts) and produces a clean `wtb_data.xlsx`. Run once, or whenever the source changes. |
| `wtb_data.xlsx` | The consolidated data: 4 sheets (`topics`, `phase_params`, `persons`, `config`). This is what the engine reads. Hand-editable. |
| `workload_engine.py` | The engine. A single `Engine` class with all calculations: synthesis at `t`, weekly/monthly forecast, peak analysis, smoothing suggestions, decisions, export. |
| `wtb_workload.ipynb` | The brain. Jupyter notebook that loads the engine, displays tables, plots curves and heat-maps, simulates transfers, and exports results. |
| `wtb_results.xlsx` | The engine output. Generated by the notebook (cell 11) or by `python workload_engine.py`. This is what a future front-end would consume. |

## Pipeline

```
  WTB_NEW__1_.xlsx (raw source)
        │  build_consolidated_data.py
        ▼
  wtb_data.xlsx (clean: topics + phase_params + persons + config)
        │  Engine (workload_engine.py)
        ▼
  Synthesis_now · Forecast · Peak · Smoothing · Decisions
        │
        ▼
  wtb_results.xlsx (export, ready for the front-end)
```

## Formulas

Per the spec:

```
VH (volume horaire, h/sem) = livrables × durée / délai
Workload = VH / weekly_capacity        (1.0 = full-time week)
```

A phase-instance is "active in week W" when:
- `status != VAL`
- end date is not stale (more than `stale_threshold_weeks` before today)
- the `[effective_start, effective_end]` window overlaps W, where:
  - `effective_start = max(today, scheduled_start)`
  - `effective_end   = max(effective_start + catchup_window, scheduled_end)`

Overdue tasks are spread over the catch-up window so they don't pile into a
single week.

## Quick start

```bash
# (One-time) refresh data from raw source
python build_consolidated_data.py

# Open the notebook
jupyter notebook wtb_workload.ipynb

# Or run engine from CLI
python workload_engine.py wtb_data.xlsx wtb_results.xlsx
```

## Adapting to other Working Units (PACK DEV, REBUY, …)

The engine is data-driven — phase parameters, persons and config all live in
the consolidated workbook. To handle a new WU:

1. Tweak `build_consolidated_data.py` to point at the WU's source file (the
   per-CA sheet structure is similar across WUs).
2. Run it to produce a new `pack_dev_data.xlsx` (or whatever name).
3. `Engine('pack_dev_data.xlsx')` — no engine change needed.

## Configuration knobs (in `wtb_data.xlsx` → `config` sheet)

| Key | Meaning | Default |
|---|---|---|
| `today` | Anchor date for the forecast | 2026-04-24 |
| `weekly_capacity_hrs` | Full-time week | 40 |
| `overload_threshold` | Workload fraction triggering the OVERLOADED label | 1.0 |
| `underload_threshold` | Workload fraction triggering the underloaded label | 0.4 |
| `catchup_window_weeks` | Min spread for overdue tasks | 4 |
| `stale_threshold_weeks` | Phases older than this with status≠VAL are excluded | 12 |
| `forecast_horizon_weeks` | How many weeks to forecast forward | 78 |

## Engine API

```python
from workload_engine import Engine

eng = Engine('wtb_data.xlsx')

eng.topics              # DataFrame: 1 row per topic (raw, from source)
eng.phase_params        # DataFrame: 5 rows (advanced/digital/static/dynamic/delivery)
eng.persons             # DataFrame: 14 CAs with active flag
eng.config              # dict

pi    = eng.build_phase_instances()   # 1 row per (CA × topic × phase)
sn    = eng.synthesis_now()           # dict: livrables, VH, summary
wkly  = eng.forecast_weekly()         # weeks × CAs
mthly = eng.forecast_monthly()        # months × CAs
peak  = eng.peak_analysis()           # KPIs per CA
smooth = eng.smoothing_suggestions()  # transfer recommendations
dec   = eng.decisions()               # action list
eng.export_results('wtb_results.xlsx')
```

## Front-end notes (for later)

The intended consumer is the export `wtb_results.xlsx` — every sheet is a flat
table ready to be loaded as a dashboard datasource. Or, since the engine is
pure Python with no GUI deps, a Streamlit/Dash front-end can call it directly:

```python
from workload_engine import Engine
eng = Engine('wtb_data.xlsx')
forecast = eng.forecast_weekly()
# ... plot in your front-end framework
```
